개념 잡기

스킬(Skill)이란?

AI 코딩 에이전트에게 “이런 종류의 일은 이렇게 다뤄”라고 알려주는, 재사용 가능한 지침 한 묶음. 각 스킬은 SKILL.md 한 장이면 끝.

Small

작다

한 가지 규율만 담는다. 한 번에 하나씩 로드되어, 필요한 것만 켠다.

Composable

조합된다

스킬이 다른 스킬을 이름으로 부른다. 발견 → 구체화 → 계획 → 구현으로 이어진다.

Model-agnostic

모델 독립적

특정 모델에 묶이지 않는다. 원하는 것만 설치해 내 것으로 고쳐 쓴다.

방법(method)이 아니라, 목표·제약·완료 조건을 적는다.AGENTS.md — Skills stay thin

전체 지도파이프라인 개요SHEET 03 / 15

관통 원칙 · 하나

모든 움직임은 “초안(Draft)”이다

빈 질문을 던지지 않는다. 고칠 수 있는 구체적 후보를 먼저 내민다. 진행의 단위는 사용자의 수정.

형태 1

가정 (veto 가능)

어긋날 가능성이 낮거나 고치기 싼 결정은, 내가 정하고 “거부권 아래 가정”으로 밝힌다.

형태 2

추천답이 붙은 질문

틀리면 비싼 갈림길만, 추천 답과 함께 한 턴에 하나씩 묻는다.

형태 3

렌더된 변형(variant)

보고 판단할 문제는 두세 개의 렌더된 안으로. 반응이 곧 답.

형태 4

구조 다이어그램

말로 두 번 이상 설명해야 할 구조는, 이해를 그림으로 되비춘다.

사람은 빈 종이를 채우기보다, 초안을 훨씬 더 잘 고친다.shape-idea — every move is a draft

관통 원칙 · 둘

스킬은 얇게 유지한다

스킬은 목표·제약·완료 조건만 적는다. 정해진 절차(procedure)는 반복해서 관찰된 실패를 막을 때만 자리를 얻는다.

담는 것

목표 · 제약 · 완료 조건

“무엇을 이뤄야 하고, 무엇을 건드리면 안 되며, 언제 끝났는가.” 방법은 모델에게 맡긴다.

아끼는 것

고정된 절차

예외적으로만 — build-prototype의 “뼈대 먼저, 채우기 나중”처럼 실패가 반복될 때만.

모델이 좋아질수록, 지침은 얇아진다.decision 0009 — thin skills over fixed procedures

언제

받침스택 컨텍스트 설치SHEET 12 / 15

무엇이 남는가

산출물 — 스펙 폴더

작업 한 단위가 남기는 것은 docs/specs/<slug>/ 폴더 하나. 결정이 앵커, 시각물은 (승인된 프로토타입만 빼고) 버려진다.

<repo>/ ├─ GLOSSARY.md # 프로젝트 공용 언어 (ubiquitous language) └─ docs/ ├─ specs/<slug>/ # 작업 단위별 핸드오프 폴더 │ ├─ spec.md # 앵커 · 확정된 결정 (shape-idea) │ ├─ plan.md # 계획했다면 (draft-plan) │ └─ prototype.html # 화면을 승인했다면 (build-prototype) └─ decisions/ # 되돌리기 어려운 결정 기록

나중 구현 세션은 이 폴더만으로 일한다 — spec은 무엇을 정했는지, prototype은 어떻게 생겼는지를 말한다.
