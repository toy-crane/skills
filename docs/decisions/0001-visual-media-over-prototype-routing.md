# Clarify settles experiential questions through environment-provided visual media, not /prototype

Clarify used to route experiential questions to `/prototype`, which builds file-based
disposable artifacts, a mode switch and a build cost that fight the interview's cadence.
We now settle them (and mirror complex structure for shared understanding) through
whatever visual medium the running environment provides, described by capability and
never by tool name, because clarify is a published skill that runs on surfaces we don't
control (terminal, desktop app, web). A rendered reaction also compresses several prose
questions into one turn, which serves the skill's question-reduction goal directly.

Rules that keep the openness disciplined: pick the cheapest medium sufficient for the
question and escalate fidelity only when the question itself demands it; when no
available medium can settle a question, defer it explicitly as a remaining risk instead
of approximating; visuals are disposable scaffolding, and decisions are the deliverable.

## Considered Options

- **Keep the /prototype routing** (rejected): build weight and skill hand-off mid-interview.
- **Name concrete tools like an inline widget API** (rejected): binds the skill to one surface.
- **Approximate every question inline regardless of fidelity** (rejected): reactions to an
  insufficient medium recorded as confirmed decisions produce false alignment.

## Consequences

The `prototype` skill stays in the repo for direct invocation; only clarify's routing is
removed. Clarify's issue-artifact delivery clause is dropped, and evals #3, #4, #6, #7
must be rewritten around the new medium language.
